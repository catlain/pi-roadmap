# planPath 机制改造 + grill-me 式信息充分性评估

> 所属 Epic: E6 重构 roadmap 项目级数据为「读取时合并」模式
> 创建日期: 2026-05-30

## 一、目标与验收标准

### 目标
1. 让 planPath 机制与实际工作流匹配——planPath 是讨论结论的沉淀，不是做事前的门票
2. 加入 grill-me 式信息充分性评估——讨论不够时 AI 必须追问，不能假装理解了就拆任务
3. 修复空 planPath 问题——展示时验证文件存在性，写入后警告缺失文件

### 验收标准
- [ ] plan-description.md 有第 0 步信息充分性评估，三个维度检查（目标/方案/边界）
- [ ] decompose-epic/story.md 不再强制要求计划文档，改为"讨论充分时创建"
- [ ] roadmap_plan 写入后扫描 planPath，不存在的列出来警告
- [ ] roadmap_show/list 展示时验证 planPath 文件存在性（⚠️ 未创建）
- [ ] doing 时不再强制检查 planPath（移除门票机制）
- [ ] add_epic/add_story/add_task 加信息充分性评估提示
- [ ] 全量测试通过

## 二、实现步骤

### T1: plan-description.md 改造
- 在"第 1 步：读取现状"之前插入"第 0 步：信息充分性评估"
- 三个维度：目标是否明确、方案是否确定、边界是否清晰
- 不够→追问（一次一个问题），够→继续拆解
- 除非用户明确说"看着办"或"先拆着"

### T2: decompose-epic/story.md 改造
- 移除"每个 Epic/Story **必须**有计划文档"
- 改为"讨论充分时，用 write 创建计划文档并设置 planPath"
- 讨论简短、description 够用时，不需要 planPath

### T3: tools-plan.ts 写入后扫描
- roadmap_plan 写入成功后，遍历 JSON 中所有 planPath
- 对每个 planPath 调用 resolvePlanPath 拿绝对路径，检查 fs.existsSync
- 不存在的列在返回结果里警告 AI："以下计划文档尚未创建，请用 write 创建"

### T4: tools-query-format.ts 展示验证
- formatRoadmapDetail 新增可选参数 planPathExists?: (planPath: string) => boolean
- 有值时展示 📋，无值时展示 ⚠️ 未创建
- 调用方（show/list/search）传入带 project 的检查闭包

### T5: tools-atomic 改造
- 移除 roadmap_update doing 时的 planPath 检查提示
- add_epic/add_story/add_task 的返回结果追加信息充分性评估提示

### T6: 测试更新
- 更新已有 planPath 相关测试适配新行为
- 新增：plan 写入后警告缺失文件、展示验证、doing 不再检查

## 三、关键决策

1. **planPath 有值→文件必须存在** 是不变量，但不在流程上强制（不做门票），而是在展示层标注
2. **grill-me 评估力度**：必须追问，除非用户说"看着办"
3. **评估范围**：所有 add_* 工具和 roadmap_plan 都要评估
